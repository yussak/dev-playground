# タスク優先度

モジュラモノリス化の練習素材として「複雑さを増やす」ためのタスク一覧。
新機能追加 → モジュール分離の順で進める方針のうち、前段（複雑さ追加）に該当する。

## 優先度 ★★★★★

- 在庫 (Inventory) 機能追加
  - 注文時の減算 / キャンセル時の戻し / 在庫切れ判定
  - Product と別概念として持たせる
- 決済 (Payment) 機能追加
  - 現状 `create` 即 `confirmed` で金銭の動きが無い
  - `pending → paid` の流れを入れて Order と連動させる
- Order 状態拡張（state machine）
  - 現状 `confirmed` / `cancelled` の2値のみ
  - `pending → paid → preparing → shipped → delivered → completed` + 各経路でキャンセル/返品
  - 遷移ガード・副作用（在庫戻し・通知発火・返金）
- クーポン拡張
  - 現状 `product_id` UNIQUE で商品1対1のみ
  - 全商品適用 / カテゴリ適用 / 最低購入金額 / 併用可否 / 利用回数上限 / 配布枚数上限

## 優先度 ★★★★

- 配送 (Shipping) 機能追加
  - 住所 / 配送方法 / 追跡番号 / 状態管理
- 返品/返金
  - 既存キャンセルの逆フロー強化
  - 返品理由 / 返金額計算 / 在庫戻し
- 通知 (Notification) 機能追加
  - 注文確定 / 発送 / キャンセル / 在庫切れのメール・プッシュ
- 出品者ダッシュボード
  - 売上集計 / 販売履歴 / 期間フィルタ
- 価格計算の複雑化
  - 税 / 送料 / 複数割引の適用順（クーポン → ポイント → キャンペーン等）
  - 現状 OrdersController に直書きで薄い
- 商品カテゴリ・タグ
  - クーポンのカテゴリ適用や検索の素材になる

## 優先度 ★★★

- レビュー (Review)
  - 星評価 / コメント / 購入者限定判定
- ポイント/ロイヤリティ
  - 注文確定で付与 / 決済で利用（決済が先）
- 検索
  - 条件 / ソート / ファセット
- カートのマージ（未ログイン → ログイン後）
  - 匿名カートを session/cookie で持ち、ログイン時にマージ
- 同時実行制御（在庫が入ってから）
  - 在庫減算の楽観/悲観ロック
  - クーポン配布枚数上限の競合
- 注文の二相処理（在庫が入ってから）
  - カートで予約 → 注文確定で減算確定 / 未確定はTTLで戻し

## 優先度 ★★

- お気に入り
  - 単純な多対多。複雑さの伸びは小さい

## メモ

- ★★★★★ の4つは互いに引っ張り合うので、どれか1つ入れると他も自然と厚くなる
- 単発で完結しがちなもの（レビュー / お気に入り / 検索）は複雑さの増分が小さい
