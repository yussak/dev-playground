# モジュラモノリス練習ブランチの運用(WIP)

このリポジトリでは、モジュラモノリス（MM）移行を複数パターン試すことを目的に、専用のブランチ運用を採用する。

## ブランチ構成

- main: モノリスのまま機能開発を続ける本流。新機能は必ず main に作る。
- mm/xxx: その時点の main から切って MM 移行を試すブランチ。命名例: `mm/first-try`, `mm/event-driven`, `mm/fine-grained` など、試したいパターンが分かる名前にする。

## 運用ルール

### 新機能の実装場所

- 新機能の実装作業は main で行う。
- mm ブランチに直接機能追加はしない。mm ブランチは「main にある機能を MM 化するパターンを試す場所」と位置づける。

### main から mm への取り込み

- mm 側で新しい main の機能を扱いたくなったタイミングで取り込む。
- 無理に追従しない。main に機能が追加されるたびに同期する必要はない。常に最新 main に揃える義務はない。

### mm ブランチが古くなったとき

- main からの取り込みコスト（コンフリクト解消、既に引いた境界の引き直しなど）が MM 練習そのものより重くなりそうなら、新しい mm ブランチを最新 main から切り直す。
- 古いブランチは知見ドキュメントごと残して参考にする（消さない）。

### mm の知見

- 各 mm ブランチ内にドキュメントを置く。
- main 側の docs にはまとめない（ブランチごとに独立させる）。
- 残す内容例: 採用した分け方の意図、うまくいった点、失敗した点、次に試したいこと。

### mm から main への取り込み

- 「これが良い」と思う MM パターンが出たら検討する。
- 取り込み方法（マージ / 手で書き直し）は、その時の差分量で判断する。事前に決めない。

## この運用の意図

- main の機能開発と MM 練習を独立させる。
- やり直しコストを下げる（気に入らなければブランチを捨てるだけ）。
- 「シンプルな状態での移行」「複雑になってからの移行」を両方試せるようにする。
