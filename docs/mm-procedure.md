# モジュラモノリス移行の進め方

mm/xxx ブランチで MM 移行を試す際の作業手順。packwerk の標準的な段階導入手法に従う。

ブランチの運用ルールは [mm-branch-workflow.md](./mm-branch-workflow.md) を参照。

## 基本姿勢
****
MM 化は一度切ったら終わりではなく、継続的に質を上げていくプロセス。

- 最初は雑な境界でも OK。まずパックに分けて enforce を有効にすれば、依存関係が明示化されて改善の土台ができる
- パッケージ化することで依存が見えるようになり、クラス分割や責務の整理などの改善もやりやすくなる
- 「新規違反は防ぐ、既存違反は段階的に消す」の二段構えで進める（rubocop_todo.yml と同じコンセプト）
- 大規模PJでは「全パック切り終わる」「完全解消」をマイルストーンにせず、一部のパックから順次価値を出す

## 初回のパック切り出し（パックごとに繰り返す）

1. パックを1つ決めて、対象ファイルを `packs/<name>/` に物理移動する
2. `packs/<name>/package.yml` を作成し、`enforce_dependencies: true` / `enforce_privacy: true` を有効にする
3. Rails の autoload paths に `packs/<name>/app/...` を追加する
4. `packwerk update`（または `packwerk update-deprecations`）を実行し、現在の違反を `package_todo.yml` に書き出す
   - これで `packwerk check` は緑になる（既存違反は許容、新規違反は引き続き検出される）
5. テストが緑であることを確認する
6. コミットする
7. ブランチ内ドキュメントに所感を書く（採用した分け方、判断、迷い）
8. 次のパックがあれば 1 に戻る

## 継続的な改善（任意のタイミングで進める）

パック化したことで依存関係が明示化される。これを土台に、複数の軸で質を上げていく。

改善の種類:

- `package_todo.yml` に並ぶ既知違反をコードで解消する
- 境界を引き直す（パックの分け方が悪ければ再編する）
- パックを更に分割する（粒度を細かくする）
- クラスを分割する（パッケージ化で見えた責務の混在を解きほぐす）
- 依存方向を整理する（双方向参照を単方向にする等）
- public API を整える（外部公開する責務を絞る）

各改善のたびに以下の流れを繰り返す:

1. 改善対象を1件選ぶ
2. コードを修正する
3. `packwerk update` で `package_todo.yml` を更新する
4. テスト緑を確認する
5. コミットする

改善は単一方向ではなく、違反解消・境界の引き直し・クラス分割など多面的に進む。

## メモ: パック1つ作った後、どう進めるか

パックを1つ切り出した後、「このパックの違反を直す」か「次のパックに進む」か、選択肢の整理。

前提として、大規模PJでは全パックを切り終わるまで違反解消や改善を待つのは非現実的。「一部のパックだけでも MM 化の価値を引き出す」観点で評価する。

### A. このパックの違反を完全に解消してから次のパックへ

- 星2
- 各パックを完成形にしてから次に進むので、1パックずつの達成感がある
- 違反解消の手段が限られる。例えば「Product → User の依存」を解消したい場合、User がまだパック化されていないと「依存先を別パックに切り出して整理する」という対処が取れない
- 「完全解消」にこだわると進まなくなる。実務では現実的でない

### B. 違反は todo に残したまま次のパックを切り出す（全パック切り出してから違反解消）

- 星2
- 大規模PJでは「全パック切り終わる」が事実上のゴールにならない
- 「一部のパックだけでも MM 化の価値を引き出す」という観点と矛盾する
- 全体像が見えるメリットはあるが、価値発生のタイミングが遅すぎる

### C. ハイブリッド: 切ったパックの範囲で可能な改善を進めつつ、新パック切り出しも並行で行う

- 星5（推奨）
- 切ったパック単位で、できる範囲の違反解消・境界調整・クラス分割を進める
- 並行して新パックの切り出しも進む
- 「全パック切り終わる」も「完全解消」もマイルストーンにせず、価値を継続的に出していく
- 大規模PJで現実的な進め方
- 進めるペースの判断が毎回必要だが、それは大規模PJでは避けられない性質

### D. このパックの境界そのものを再考する（範囲を変える / 更に分割する）

- 星2
- 早めの調整は重要だが、判断材料が少ない段階で再考しても精度が出ない
- C で改善を進める中で必要に応じて再考する方が現実的

推奨は C。理由は「大規模PJで全パック切り終わるのを待たずに、一部のパックから価値を出していける」「違反解消・境界調整・新パック切り出しを並行で進められる」「『完全解消』『全パック切り終わる』を無理なゴールにしない」。
