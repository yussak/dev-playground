# ADR-007: BFF を Next.js 内に導入する

## ステータス

採用

## コンテキスト

フロントエンド（Next.js / TypeScript）からバックエンド（Rails API）への通信が混在している。

- 取得系（GET）: 各 Server Component が `frontend/lib/api.ts` の `apiFetch` で Rails を直叩き（`products/page.tsx`, `orders/page.tsx` など）
- 更新系の一部: `frontend/app/api/orders/route.ts`（Route Handler）を経由

このリポジトリはモジュラーモノリスの練習が目的であり、今後モジュールを分割していくと「1画面のために複数モジュールを横断集約する」場面が増える（例: カート画面 = cart + product + stock + coupon）。この集約を Rails 側でやるとモジュール間に不要な依存が生まれる。集約と画面都合の整形を担う層（BFF）が欲しい。

なお、本プロジェクトはローカルでのみ動かす前提であり、独立デプロイ・独立スケールの要求はない。

## 検討した案

### 案①: Next.js 自体を BFF にする（同一プロセス）

Next のサーバーサイド（Server Component / Route Handler）で Rails を叩く。新規プロセスを建てない。

- メリット: インフラ増分ゼロ。フロントと型共有が自動。Next/TS 構成の王道
- デメリット: フロントと物理分離されず「マイクロサービス感」は薄い

### 案②: 別プロセスで Hono（TS）

`hono` で独立サーバーを立て、compose にサービスを追加する。

- メリット: プロセス分割を体感できる。`hc` で型付き RPC も使える
- デメリット: ローカル専用だと、プロセス増・配線のコストに見合う旨味（独立デプロイ/スケール）が無い

### 案③: 別プロセスで Express（TS）

従来からの定番フレームワークで別建てする。

- メリット: 枯れていて情報が多い
- デメリット: 今から TS で新規に建てるなら Web 標準・型・軽さで Hono に劣り、あえて選ぶ理由がない

## 決定

**案①を採用。** BFF は別プロセスを建てず、Next.js の実行環境内に置く。

あわせて技術的な配置の規約を以下に定める。

- **`frontend/lib/bff/*`** が BFF の本体（サーバー専用の取得・集約関数）。Server Component はこれを **関数として直接呼ぶ**
- **`frontend/app/api/*`**（Route Handler）は **ブラウザから直接叩かれる必要があるものだけ**の窓口にする。中身は `lib/bff` に委譲して薄く保つ
- 振り分けの判断基準は「**ブラウザが直接叩く必要があるか**」
- 取得系（GET）を `app/api` に置かない。Server Component → 自分の Route Handler → Rails という**二重ホップ**を避けるため
- **ドメインロジック（割引計算・在庫引当の可否判定など）は Rails に残す**。BFF は集約と画面都合の整形に徹する

## 採用理由

- ローカル専用のため、別プロセス化の主目的（独立デプロイ・独立スケール）が存在しない。②③のコストを払う理由がない
- Next/TS 構成では①が一般的な王道。インフラ増分ゼロで、フロントと型共有が自動的に効く
- 「まず最低限動く実装から始める」という方針（CLAUDE.md）と整合する

## 影響

- `frontend/lib/api.ts` の `apiFetch` は廃止せず、`lib/bff` の内部実装（Rails を叩く低レベル関数）として温存する。画面からの直接呼び出しはやめる
- 段階移行とする: 単純な画面（products）で型・規約を固める → カート画面で横断集約を試す → 残り画面を順次移行 → `app/api/orders` を薄い委譲に整理
- テストは「振る舞い」を検証する（Rails をモックし集約結果の形を確認、`frontend/lib/__tests__` の流儀に合わせる）
- **未決**: 横断集約を BFF でやるか、Rails 側に読み取り専用の Query モデルを切るかは画面ごとに判断する。原則は BFF 集約とし、性能や境界が破綻する画面のみ Rails 側 Query を検討する（必要になった時点で別 ADR 化する）
