# ADR-007: モジュラモノリス移行の最初の一歩

## ステータス

採用

## コンテキスト

このリポジトリはモジュラモノリス（以下 MM）移行を試すための練習場であり、現状は素の Rails モノリス。Gemfile に `packwerk` は導入済みだが、`backend/package.yml` はルート package のみで何も分割されていない。

MM 化の設計判断軸（境界の引き方、public API、モジュール間通信、データ所有権、DB 分離など）は多数あるが、コードを実際に分割してみる前に机上で全部決めるのは難しい。何が痛いのかが分からない状態で深い議論をしても、判断材料が不足する。

そのため「最初の一歩」として、現実的に動かせる最小限のスコープで一度試行し、得られた知見をもとに次の判断に進む方針を取る。

## 決定

最初の試行として、以下のスコープで MM 化を試す。

### 1. パック分割

ドメインベースで 6 パックに荒く分ける。

| パック | 含むモデル |
|---|---|
| `iam` | User, Admin |
| `catalog` | Product, ProductImage, ProductVariant |
| `inventory` | Stock |
| `cart` | Cart, CartItem |
| `order` | Order, OrderItem |
| `coupon` | Coupon, CouponUse |

切る順序は `iam` → `catalog` → 残り。`iam` は依存先が無い葉、`catalog` は他全パックが依存する中心ノードのため先に切る。

### 2. 強制レベル

- `enforce_dependencies: true`（依存方向を可視化・違反検出）
- `enforce_privacy: false`（public API 整備は今回のスコープ外）

### 3. 既存違反の扱い

`packwerk update` で `package_todo.yml` に逃がして緑化する。違反解消は今回のスコープ外。新規違反のみ検出される状態を作る。

### 4. スコープ外（意図的にやらない）

- public API（`app/public/` 整備）
- テストコードの `packs/<name>/spec/` への移動
- モデル間関連の ID 参照化
- イベント駆動・PubSub 導入
- DB スキーマ/インスタンス分離

これらは試行の過程で必要性を判断し、必要なら別 ADR で決定する。

### 5. 進め方

`docs/mm-procedure.md` の方針 B（全パック切り出し → 違反は `package_todo.yml` に逃がす → 段階解消）を採用する。1 パック完成型ではなく、全体を荒く切ってから改善する。

### 6. ブランチ運用

`docs/mm-branch-workflow.md` に従い、main から `mm/xxx` ブランチを切って試す。試行で得た知見はブランチ内ドキュメントに残す。

## 理由

- **試行優先**: 何が痛いのか分からない段階での机上設計は判断材料が不足する。まず動かす
- **6 パックは現状のドメイン構造に素直**: モデルとコントローラの観察から自然に切れる粒度
- **`enforce_privacy` を外す理由**: 公開 API を最初から強制するとパック切り出し自体のコストが跳ね上がり、初手で潰れる。「依存方向が見える」だけを最初のゴールにする
- **`package_todo.yml` で逃がす理由**: 違反解消とパック切り出しを同時にやると、依存先パックが未分割で手詰まりになる。`rubocop_todo.yml` と同じ二段構え
- **スコープを意図的に絞る理由**: MM 化は継続プロセス。最初の試行で全部やろうとすると、何が効いたのか分からなくなる

## 影響

- `packs/<name>/` 配下にコードが物理移動する
- `packwerk check` がプロジェクトの一部に組み込まれる（CI 統合は別途判断）
- public API・データ所有権・モジュール間通信などの判断は試行中に随時 ADR を追加する
- この ADR は「最初の一歩」の記録であり、より良いパターンが見つかれば別 ADR で更新する
